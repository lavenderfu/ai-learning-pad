## 完整示例

以下是一个「校园二手交易」微信小程序的 Product Spec 示例，供参考：

```markdown
## 产品概述

这是一个面向大学生的校园二手物品交易小程序，让学生在毕业季或日常生活中快速发布和购买二手物品。

**目标用户**：在校大学生，尤其是毕业季需要处理闲置物品的学生，以及想低价淘到实用物品的新生。痛点是「闲鱼上校内交易不方便，微信群里信息太乱找不到想要的东西」。

**核心价值**：基于微信社交关系和地理位置，让校内二手交易更高效、更可信。AI 辅助定价和描述生成，降低发布门槛。

**产品形态**：微信小程序

## 应用场景

- **毕业清仓**：大四学生小张毕业要处理一堆课本和生活用品，拍几张照片上传，AI 自动生成描述和建议定价，5 分钟发布了 10 件商品。
- **新生入学**：大一新生小李刚入学，想买二手自行车和台灯，打开小程序按分类筛选，看到本校学姐发的商品，微信直接联系约面交。
- **日常置换**：小王买了新耳机想把旧的出掉，发布后同楼栋的同学看到了，下楼就完成了交易。

## 功能需求

**核心功能**
- 商品发布：用户上传照片 → 点击「AI 生成描述」→ AI 自动识别物品类别并生成描述和建议价格 → 用户确认后发布
- 商品浏览：用户在首页 → 按分类/价格/距离筛选 → 浏览商品卡片列表
- 商品详情：用户点击商品卡片 → 查看详情（图片、描述、价格、卖家信息）→ 点击「联系卖家」通过微信聊天沟通
- 搜索：用户在搜索栏输入关键词 → 系统返回匹配商品列表

**辅助功能**
- 我的发布：查看和管理自己发布的商品（上架/下架/删除）
- 收藏：收藏感兴趣的商品，在「我的收藏」中查看

## UI 布局

### 整体导航
底部 TabBar 3 个入口：首页、发布（+号）、我的

### 首页
- 顶部：搜索栏
- 分类标签栏：全部 / 教材 / 电子产品 / 生活用品 / 其他
- 商品列表：瀑布流卡片布局，每张卡片包含：封面图、标题、价格、发布者头像

### 发布页
- 图片上传区：最多 9 张
- 「AI 生成描述」按钮
- 标题输入框
- 描述文本框（AI 生成后可编辑）
- 价格输入框（带 AI 建议价格提示）
- 分类选择：下拉框
- 「发布」按钮

### 我的页
- 用户信息卡片
- 功能入口列表：我的发布、我的收藏

## 用户使用流程

### 发布商品
1. 点击 TabBar「发布」
2. 上传商品照片
3. 点击「AI 生成描述」
4. 确认或编辑标题、描述、价格
5. 选择分类
6. 点击「发布」

### 购买商品
1. 在首页浏览或搜索商品
2. 点击商品卡片查看详情
3. 点击「联系卖家」
4. 通过微信聊天沟通，线下面交

## AI 能力需求

| 能力类型 | 用途说明 | 应用位置 |
|---------|---------|---------|
| 图像识别 | 识别用户上传的商品照片，判断物品类别 | 发布页点击「AI 生成描述」时 |
| 文本生成 | 根据图片识别结果生成商品标题和描述 | 发布页点击「AI 生成描述」时 |
| 智能定价 | 根据物品类别和成色建议合理价格 | 发布页 AI 生成描述后自动填充建议价格 |

## 小程序页面/组件规划

| 页面/组件名称 | 用途 | 核心功能 | 技术实现 | 导航/用户流程 | 文件路径 |
|:--------:|:----:|:--------:|:--------:|:--------:|:--------:|
| 首页 | 商品浏览 | 分类筛选、商品列表、搜索 | wx.cloud.callFunction 获取列表 | TabBar 入口 | pages/index/index |
| 发布页 | 发布商品 | 图片上传、AI 生成、表单提交 | wx.chooseImage + 云函数调用 AI | TabBar「+」入口 | pages/publish/publish |
| 详情页 | 商品详情 | 图片轮播、信息展示、联系卖家 | navigateTo 跳转 | 首页点击卡片 | pages/detail/detail |
| 我的页 | 个人中心 | 我的发布、我的收藏 | wx.getUserProfile 获取用户信息 | TabBar 入口 | pages/mine/mine |
| 商品卡片 | 公共组件 | 展示商品缩略信息 | 自定义组件 | 首页引用 | components/goods-card/goods-card |

## 后端接口设计

| 接口名称 | 用途 | 请求方法 | 请求路径 | 请求参数 | 返回数据 | 实现位置 |
|:--------:|:----:|:--------:|:--------:|:--------:|:--------:|:--------:|
| getGoodsList | 获取商品列表 | POST | /getGoodsList | { category, page, pageSize } | { list, total } | 云函数 |
| publishGoods | 发布商品 | POST | /publishGoods | { title, desc, price, images, category } | { goodsId } | 云函数 |
| getGoodsDetail | 获取商品详情 | POST | /getGoodsDetail | { goodsId } | { goods } | 云函数 |
| aiGenerateDesc | AI 生成描述 | POST | /aiGenerateDesc | { imageFileIDs } | { title, desc, suggestPrice } | 云函数 |

## 微信开放能力需求

| 能力名称 | 用途说明 | 应用位置 |
|---------|---------|---------|
| 登录授权 | 获取用户 openid，关联发布数据 | 首次进入时静默登录 |
| 用户信息 | 获取头像昵称，展示在卖家信息中 | 「我的」页面触发 |
| 图片上传 | 选择和上传商品照片到云存储 | 发布页 |
```

## 写作要点

1. **产品概述**：
   - 一句话说清楚是什么
   - **必须明确写出目标用户**：是谁、有什么特点、什么痛点
   - 核心价值：用了这个小程序能得到什么
   - **必须标注产品形态为微信小程序**

2. **应用场景**：
   - 具体的人 + 具体的情况 + 具体的用法 + 解决什么问题
   - 场景要有画面感，让人一看就懂

3. **功能需求**：
   - 分「核心功能」和「辅助功能」
   - 每条格式：用户做什么 → 系统做什么 → 得到什么
   - 写清楚触发方式

4. **UI 布局**：
   - 先写导航方式（TabBar / 非 TabBar）
   - 再逐页描述布局和控件
   - 控件要具体：下拉框写出所有选项和默认值

5. **小程序页面/组件规划**：
   - 每个页面写清楚：用途、核心功能、技术实现方式、文件路径
   - 公共组件单独列出

6. **后端接口设计**：
   - 每个接口写清楚：请求方法、参数、返回格式、实现位置
   - 如不需要后端，标注「无需后端服务」

7. **微信开放能力需求**：
   - 列出所有需要的微信 API
   - 写清楚在哪个环节触发

8. **开发状态跟踪**：
   - 初始全部标注「未开始」
   - 开发过程中自动更新状态
