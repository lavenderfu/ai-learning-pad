# 阶段2：核心搭建 协议（微信小程序）

本协议定义了阶段2（核心搭建）针对微信小程序的完整工作流程。阶段2的目标是将阶段1的产品规格文档和原型页面转化为可在微信开发者工具中运行的小程序项目。

---

## 角色职责

| 角色 | 定位 | 职责 |
|------|------|------|
| **架构师** | 主导 | 架构设计、前后端开发、云函数开发、AI 能力集成、Code Review |
| **PM（产品经理）** | 辅助 | 验收核心流程、确认功能与 spec 一致 |

### 角色行为差异

- **当前角色为架构师时**：执行完整的架构设计和四子阶段开发流程，做出技术决策，创建项目级 CLAUDE.md
- **当前角色为 PM 时**：对照 spec 验收核心流程，确认功能实现是否符合需求文档定义，提出验收意见

---

## 进入条件

先执行阶段1完成度检查：

1. 读取 `1-prototype/spec.md`，确认以下内容完整：
   - 产品定位、目标用户、功能清单、用户故事、验收标准、AI 能力需求
   - 小程序页面/组件规划表
   - 后端接口设计表
   - 微信开放能力需求
2. 检查 `1-prototype/pages/` 目录，确认原型可参考
3. 如有缺失，列出缺失项并提醒用户补充，**不开始开发**

## 工作流程

### 1. 审阅阶段1产出

读取以下文件，全面理解产品需求：

- `1-prototype/spec.md`：产品规格文档，提取：
  - 产品概述、功能需求、UI 布局
  - 小程序页面/组件规划
  - 后端接口设计
  - 微信开放能力需求
  - AI 能力需求
  - 开发状态跟踪表
- `1-prototype/pages/`：原型页面，了解核心交互流程和视觉设计参考

如果存在以下可选文件，也一并读取：

- `1-prototype/ui-prompts.md`：UI 原型图提示词，参考视觉设计方向
- `1-prototype/competitive.md`：竞品分析，了解差异化定位

---

### 2. 创建项目级 CLAUDE.md

- **模板**：`templates/project-CLAUDE-template.md`
- **输出位置**：`2-build/CLAUDE.md`
- **内容**：定义本项目的技术栈、代码规范、目录结构、开发命令

根据模板填写以下信息：

- 技术栈（微信小程序原生框架、云开发/自建后端、AI 能力）
- 目录结构（小程序标准目录结构）
- 代码规范（WXML/WXSS/JS 规范、命名规范）
- 开发命令（微信开发者工具操作指南）
- 项目特殊注意事项

---

### 3. 四子阶段开发流程

小程序开发分为四个子阶段，按顺序执行：

```
前端开发 → 后端开发 → 联调 → 测试
```

#### 3.1 前端开发阶段

##### 确定开发范围

- **批量模式**：所有未开发的前端页面
- **单页模式**：仅指定的单个页面

检查页面是否在 spec 的页面/组件规划表中：
- 不存在 → 提示用户并列出可开发的页面
- 已完成 → 询问是否需要重新开发

##### 对每个页面执行技术方案设计

a. **UI 设计方案**：
   - 从 `1-prototype/spec.md` 读取 UI 风格偏好，按其要求执行
   - 遵循微信小程序设计指南
   - 页面布局：考虑内容优先级，合理划分区域
   - 交互体验：动画过渡自然，响应迅速，反馈明确
   - 组件选择：优先使用原生组件

b. **交互实现**：详细说明页面规划中定义的导航/用户流程，确定 `wx.navigateTo`/`wx.switchTab`/`wx.reLaunch` 等具体导航方式

c. **组件复用**：识别可抽离的公共组件，规划在 `/components/` 目录下的实现

d. **性能优化策略**：针对该页面特点规划优化方向

e. **功能完整性检查表**：基于页面规划中的核心功能，列出详细功能点清单

输出技术方案后确认：「已完成 [页面名称] 的技术方案，现在基于此方案生成代码。」

##### 页面代码生成（基于技术方案）

- 创建 WXML、WXSS、JS、JSON 文件
- 实现页面布局、样式和逻辑
- 使用模拟数据实现界面交互（针对需要后端接口的功能）
- 确保代码符合微信小程序规范：
  - 架构规范：使用官方框架，正确组织文件结构
  - 性能优化：优化 setData 调用，实现懒加载
- 确保设计符合 UI/UX 最佳实践
- 考虑不同设备和屏幕尺寸的适配性

##### 更新开发状态

更新 `1-prototype/spec.md` 中「开发状态跟踪」表，将对应页面状态改为「已完成」，通知开发进度。

---

#### 3.2 后端开发阶段

##### 检查前端开发状态

读取 spec 中的开发状态跟踪表：
- 前端页面未全部完成 → 建议先完成前端：「还有未完成的前端页面：[列表]。建议先完成前端开发后再进行后端开发。」
- 前端已全部完成 → 继续

##### 询问开发文档

「您是否有现成的后端开发文档？如有的话可以发给我，包括但不限于 API 调用文档、开发文档链接、数据库文档等。如没有，请继续。」

##### 执行后端接口开发

确定要开发的接口列表：spec 中后端接口设计表里所有未开发的接口

对每个接口执行：

a. **接口设计**：
   - 根据前端页面的需求，明确接口功能和数据格式
   - 设计参数校验、返回结构、错误处理和权限控制
   - 参考微信云开发最佳实践

b. **接口实现**：
   - 创建云函数代码（index.js + package.json）
   - 实现参数校验、业务逻辑、数据库操作和错误处理
   - 提供测试示例

c. **更新状态**：
   - 更新 spec 中开发状态跟踪表，接口状态改为「已完成」
   - 通知进度

---

#### 3.3 联调阶段

##### 检查开发状态

读取 spec 中的开发状态跟踪表：
- 存在未开发页面或接口 → 建议先完成开发：「还有未完成的页面或接口：[列表]。建议先完成全部开发工作再进行联调。」
- 已全部开发完成 → 继续

##### 确认联调范围

- **全部联调**：所有页面和接口
- **指定页面联调**：指定页面及其依赖接口

##### 执行联调

a. **页面级联调**：
   针对每个需要调用后端的页面：
   - 将模拟数据替换为真实接口调用
   - 验证请求参数的正确性
   - 验证返回数据的处理逻辑
   - 验证错误处理机制
   - 每完成一个页面通知进度

b. **流程级联调**：
   测试完整用户流程，按用户实际使用路径验证：
   - 页面间正确跳转
   - 数据在多页面间的传递和使用
   - 用户会话状态的维护
   - 边界条件处理

##### 更新状态

更新 spec 中开发状态跟踪表，联调状态改为「已完成」。

---

#### 3.4 测试阶段

##### 确认测试范围

- **完整测试**：整个小程序的完整测试用例集
- **专项测试**：指定页面的专项测试用例

##### 创建测试用例

保存至 `2-build/tests/` 目录，包含：

a. **功能测试**：验证页面所有功能是否按预期工作

b. **界面测试**：验证 UI 元素是否正确显示

c. **交互测试**：验证用户交互流程是否顺畅

d. **边缘情况测试**：测试异常情况处理（网络中断、权限拒绝等）

e. **性能测试**：页面加载时间、操作响应速度等

##### 云函数测试

如使用云函数，创建云函数调用测试示例。

---

### 4. 环境变量安全规范

#### 安全警告

- **涉及 AI API 调用时**：API Key 必须放在云函数环境变量中，**绝对不可暴露给前端**
- **小程序前端代码会被微信解包**：任何写在前端代码中的密钥都会泄露

#### 文件规范

- 创建 `.env.example` 作为模板（提交到 Git），列出所有需要的环境变量名和注释说明
- 实际值放在云函数的环境变量配置中（通过微信开发者工具配置）

---

### 5. 代码规范

#### 文件规范

- 单个文件不超过 300 行，超过则拆分
- 项目结构必须符合小程序标准：`/pages`、`/utils`、`/components`、`/static`、`app.js|json|wxss`

#### 项目结构

```
2-build/
├── miniprogram/
│   ├── app.js                    # 小程序入口
│   ├── app.json                  # 全局配置
│   ├── app.wxss                  # 全局样式
│   ├── pages/                    # 页面目录
│   │   └── <页面名>/
│   │       ├── <页面名>.wxml
│   │       ├── <页面名>.wxss
│   │       ├── <页面名>.js
│   │       └── <页面名>.json
│   ├── components/               # 公共组件
│   ├── utils/                    # 工具函数
│   └── static/                   # 静态资源
├── cloudfunctions/               # 云函数（如使用云开发）
│   └── <函数名>/
│       ├── index.js
│       └── package.json
└── project.config.json           # 项目配置
```

#### 依赖选择原则

- 只装需要的，不装「可能用到」的
- 优先使用微信原生组件
- 只改与当前任务相关的文件

#### 通用规范

- 代码要简洁、可读、可维护
- 优先使用简单方案，不过度设计
- 每个阶段完成后输出进度反馈
- 如有原型图，开发时参考原型图的视觉设计

---

### 6. 质量门禁

每个功能点至少满足以下标准：

#### 必须满足

- [ ] 代码可直接在微信开发者工具中运行
- [ ] 符合微信小程序规范和最佳实践
- [ ] 主路径可用（Happy Path 能跑通）
- [ ] 异常路径清晰（错误提示、重试/回退）
- [ ] loading 状态（涉及异步操作时）
- [ ] 空状态处理（无数据时的提示）
- [ ] 基础输入校验（必填、格式）
- [ ] 敏感信息不写入代码（API Key 走云函数环境变量）

#### 建议满足

- [ ] 基础可访问性（可点击、可键盘操作）
- [ ] 不同设备屏幕适配
- [ ] 优化 setData 调用频次和数据量

- [ ] 项目可在微信开发者工具中运行

- [ ] 后端接口全部完成（对照 spec 中的后端接口设计表）
- [ ] 联调完成，数据流转正常
- [ ] 质量门禁的「必须满足」项全部通过
- [ ] API 文档已生成（`2-build/docs/api.md`）
- [ ] 交接文档已写明完成项和遗留项（`2-build/docs/handover.md`）
- [ ] 项目级 CLAUDE.md 已创建（`2-build/CLAUDE.md`）
- [ ] PM 验收核心流程通过
- [ ] `stage.md` 中 `current_stage` 更新为 `3`
- [ ] `1-prototype/spec.md` 中开发状态跟踪表已更新

当用户说"阶段2完成"或要求进入阶段3时，逐项检查上述清单，缺失则列出缺失项并提醒补充，**不允许跳过门禁直接进入下一阶段**。
