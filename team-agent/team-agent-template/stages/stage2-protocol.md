# 阶段2：核心搭建 协议（Web 应用）

本协议定义了阶段2（核心搭建）针对 Web 应用的完整工作流程。阶段2的目标是将阶段1的产品规格文档和原型页面转化为可运行的项目代码，核心流程跑通，功能可用。

---

## 角色职责

| 角色 | 定位 | 职责 |
|------|------|------|
| **架构师** | 主导 | 架构设计、技术选型、核心开发、AI/智能体集成、Code Review |
| **PM（产品经理）** | 辅助 | 验收核心流程、确认功能与 spec 一致 |

### 角色行为差异

- **当前角色为架构师时**：执行完整的架构设计和核心开发流程，做出技术决策，创建项目级 CLAUDE.md
- **当前角色为 PM 时**：对照 spec 验收核心流程，确认功能实现是否符合需求文档定义，提出验收意见

---

## 进入条件

先执行阶段1完成度检查：

1. 读取 `1-prototype/spec.md`，确认内容完整（包含产品定位、目标用户、功能清单、用户故事、验收标准、AI 能力需求）
2. 检查 `1-prototype/pages/` 目录，确认原型可参考
3. 如有缺失，列出缺失项并提醒用户补充，**不开始开发**

## 工作流程

### 1. 审阅阶段1产出

读取以下文件，全面理解产品需求：

- `1-prototype/spec.md`：产品规格文档，提取产品概述、功能需求、UI 布局、技术说明、AI 能力需求
- `1-prototype/pages/`：原型页面，了解核心交互流程和视觉设计参考

如果存在以下可选文件，也一并读取：

- `1-prototype/ui-prompts.md`：UI 原型图提示词，参考视觉设计方向
- `1-prototype/competitive.md`：竞品分析，了解差异化定位

---

### 2. 创建项目级 CLAUDE.md

- **模板**：`templates/project-CLAUDE-template.md`
- **输出位置**：`2-build/CLAUDE.md`
- **内容**：定义本项目的技术栈、代码规范、目录结构、API 约定、开发命令

根据模板填写以下信息：

- 技术栈（前端框架、后端框架、数据库、部署方式、AI 能力）
- 目录结构（根据项目类型调整）
- 代码规范（语言、代码风格、命名规范、Git 分支策略）
- API 约定（基础路径、响应格式、认证方式）
- 开发命令（启动、测试、构建）
- 项目特殊注意事项

---

### 3. 项目类型判断

根据 spec 的 UI 布局和技术说明判断项目类型：

| 判断条件 | 项目类型 | 说明 |
|---------|---------|------|
| 产品形态标注为「微信小程序」 | 微信小程序 | 提示使用小程序协议（加载 `stages/stage2-miniprogram-protocol.md`），终止当前协议 |
| 有 UI + 纯前端/无需服务器 | 纯前端 Web 应用 | 不需要后端，所有逻辑在浏览器运行 |
| 有 UI + 需要后端/数据库/API | 全栈 Web 应用 | 前后端一体，需要服务端处理 |
| 无 UI + 命令行操作 | CLI 工具 | 终端交互，无图形界面 |
| 只是 API 服务 | 后端服务 | 只提供接口，无前端界面 |

**路由规则**：

- 如果 spec 中产品形态标注为「微信小程序」→ 提示用户：「检测到这是微信小程序项目，将使用小程序专用开发流程。」→ 加载 `stages/stage2-miniprogram-protocol.md` → 终止当前协议
- 其他类型 → 继续执行当前协议

---

### 4. 技术栈选择

#### 推荐技术栈

| 项目类型 | 推荐技术栈 |
|---------|-----------|
| 纯前端 Web 应用 | React + Vite + TypeScript + Tailwind |
| 全栈 Web 应用 | Next.js + TypeScript + Tailwind |
| CLI 工具 | Node.js + TypeScript + Commander |
| 后端服务 | Express + TypeScript |

#### 选择原则

1. spec 技术说明有指定 → 用指定的
2. 没指定 → 用推荐方案
3. 有疑问 → 询问用户

---

### 5. 环境变量安全规范

#### 安全警告

- **Vite 纯前端**：`VITE_` 前缀变量**会暴露给浏览器**，不能存放 API Key
- **Next.js**：不加 `NEXT_PUBLIC_` 前缀的变量只在服务端可用（安全）

#### 涉及 AI API 调用时

- **推荐**：用 Next.js（API Key 只在服务端使用，安全）
- **备选**：创建独立后端代理请求
- **仅限开发/演示**：使用 `VITE_` 前缀（必须提醒用户安全风险）

#### 文件规范

- 创建 `.env.example` 作为模板（提交到 Git），列出所有需要的环境变量名和注释说明
- 实际值放 `.env.local`（不提交），确保 `.gitignore` 包含 `.env.local`
- 敏感信息（API Key、数据库密码等）只能放在 `.env.local` 中

### 6. 开发流程

#### 6.1 启动阶段

> 目的：检查前置条件，读取项目文档

**第一步：检测 spec**

- 检测 `1-prototype/spec.md` 是否存在
- 不存在 → 提示：「未找到 1-prototype/spec.md，请先完成阶段1（产品定义）。」→ 终止流程
- 存在 → 继续

**第二步：读取项目文档**

- 加载 `1-prototype/spec.md`
- 提取：产品概述、功能需求、UI 布局、技术说明、AI 能力需求

**第三步：项目类型路由**

- 如果 spec 中产品形态标注为「微信小程序」→ 提示：「检测到这是微信小程序项目，将使用小程序专用开发流程。」→ 加载 `stages/stage2-miniprogram-protocol.md` → 终止当前协议
- 否则 → 继续

**第四步：检查原型图**

- 检查 `1-prototype/pages/` 目录中是否有原型页面
- 有 → 开发时参考原型页面的视觉设计和交互逻辑
- 检查 `1-prototype/ui-prompts.md` 是否存在
- 存在 → 询问：「我看到已有原型图提示词，如果有生成的原型图图片，可以提供参考。」
- 不存在 → 询问：「是否有原型图或设计稿可以参考？有的话可以提供。」
- 用户提供图片 → 记录，开发时参考
- 用户说没有 → 继续

#### 6.2 技术方案阶段

> 目的：确定技术栈并告知用户

分析项目类型，选择技术栈，列出主要依赖。

输出方案后直接进入下一阶段：

```
**技术方案**

**项目类型**：[类型]
**技术栈**：[技术栈]
**主要依赖**：
- [依赖1]：[用途]
- [依赖2]：[用途]
```

#### 6.3 项目搭建阶段

> 目的：初始化项目，创建基础结构

执行步骤：

1. **初始化项目**：根据技术栈执行对应的初始化命令
2. **配置 Tailwind**（Vite 项目）：安装和配置 Tailwind CSS
3. **安装功能依赖**：根据 spec 需求安装必要的第三方库
4. **配置环境变量**（如需要）：创建 `.env.example` 和 `.env.local`

每完成一步输出进度反馈。

**注意**：所有项目代码创建在 `2-build/` 目录下。

#### 6.4 代码实现阶段

> 目的：实现功能代码

**第一步：创建基础布局**

- 根据 spec 的 UI 布局章节创建整体布局结构
- 如有原型页面，参考其视觉设计

**第二步：实现 UI 组件**

- 根据 UI 布局的控件规范创建组件
- 使用 Tailwind 编写样式

**第三步：实现功能逻辑**

- 核心功能优先实现，辅助功能其次
- 添加状态管理，实现用户交互逻辑

**第四步：集成 AI 能力（如有）**

- 创建 AI 服务模块，实现调用函数
- 处理 API Key 读取（遵循环境变量安全规范）
- 在相应功能中集成 AI 能力

**第五步：完善用户体验**

- 添加 loading 状态
- 错误处理
- 空状态提示
- 输入校验

#### 6.5 完成阶段

> 目的：输出开发结果总结

- [功能1]

**AI 能力集成**：
- [已集成的 AI 能力，或「无」]

### 7. 代码规范

#### 文件规范

- 单个文件不超过 300 行，超过则拆分
- 优先使用函数组件 + Hooks（React 项目）
- 样式优先用 Tailwind

#### 项目名称规范

- 只能用小写字母、数字、短横线（如 `my-app`）
- 不能有空格、`&`、`#` 等特殊字符

#### npm 报错处理

- npm 初始化报错时，可尝试 pnpm 或 yarn

#### 依赖选择原则

- 只装需要的，不装「可能用到」的
- 不随手升级无关依赖
- 不做全局格式化
- 不做无关重命名
- 只改与当前任务相关的文件

#### 通用规范

- 代码要简洁、可读、可维护
- 优先使用简单方案，不过度设计
- 每个阶段完成后输出进度反馈
- 如有原型图，开发时参考原型图的视觉设计

---

### 8. 质量门禁

每个功能点至少满足以下标准：

#### 必须满足

- [ ] 主路径可用（Happy Path 能跑通）
- [ ] 异常路径清晰（错误提示、重试/回退）
- [ ] loading 状态（涉及异步操作时）
- [ ] 空状态处理（无数据时的提示）
- [ ] 基础输入校验（必填、格式）
- [ ] 敏感信息不写入代码（API Key 走环境变量）

#### 建议满足

- [ ] 基础可访问性（可点击、可键盘操作）
- [ ] 响应式适配（如需支持移动端）
